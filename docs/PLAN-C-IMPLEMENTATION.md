# RepaceClaw 方案 C 实施计划

> 创建时间：2026-05-05
> 目标：完成多账号隔离、同账号多端同步、刷新恢复、缓存治理、同步状态可视化的企业稳定版改造
> 周期预估：7~10 天
> 实施原则：后端为唯一业务真相，前端缓存只承担 UI 恢复和短时加速

---

## 一、实施目标

### 场景 1：同浏览器不同账号同时登录
- 不串 auth
- 不串会话
- 不串任务/项目/workspace

### 场景 2：同账号多 tab / 多浏览器
- 数据实时或准实时同步
- 不存在数据污染
- 同账号状态一致

### 场景 3：刷新页面
- 登录态恢复正常
- UI 状态恢复正常
- 业务数据以后端为准自动校正

---

## 二、分阶段计划

## Day 1：Auth 与 Tab 隔离层重构
### 目标
建立 tab 级 auth 隔离，切断不同账号串号根因。

### 任务
1. authStore 从 localStorage 迁移到 sessionStorage
2. 新增 tabId 生成与持久化
3. 页面顶部用户信息栏增加 userId / tabId 展示
4. 登录/退出时缓存清理逻辑标准化
5. 旧 auth 兼容迁移逻辑

### 交付物
- authStore 重构完成
- tabId 机制落地
- 用户信息栏增强

---

## Day 2：前端 Persist 分层治理
### 目标
把“业务真相”和“UI 状态缓存”彻底拆开。

### 任务
1. conversationStore 持久化内容瘦身
2. taskStore 持久化内容瘦身
3. projectKanbanStore 持久化内容瘦身
4. sessionKanbanStore 持久化内容瘦身
5. 所有 key 改为 `store:{userId}:{tabId}`
6. 业务实体改成启动后从 API 拉取，不长期本地固化

### 交付物
- 4 个 store 重构完成
- UI 状态缓存与业务数据分离完成

---

## Day 3：刷新恢复链路统一
### 目标
刷新后不串、不丢、不回流旧数据。

### 任务
1. 定义统一恢复顺序：auth → tab/ui state → backend data
2. 修复 workspace 刷新恢复链路
3. 修复 kanban 刷新恢复链路
4. 修复 session list 刷新恢复链路
5. 清理已删除数据刷新复现问题

### 交付物
- 刷新恢复机制统一
- 关键页面刷新稳定

---

## Day 4：同浏览器多 Tab 同步
### 目标
同账号多个 tab 自动同步。

### 任务
1. 封装 BroadcastChannel 同步模块
2. 建立 `rc-sync-{userId}` 频道
3. 定义前端同步事件协议
4. conversations/tasks/projects 增量广播
5. workspace / kanban / session list 联动更新

### 交付物
- 同浏览器多 tab 同步可用

---

## Day 5：多浏览器/多设备同步
### 目标
同账号在不同浏览器、不同设备保持状态一致。

### 任务
1. 后端增加 user 级同步事件通道
2. 复用/扩展当前 WebSocket
3. 前端订阅 user:{userId} 事件
4. 收到事件后按资源局部刷新
5. 增加连接状态与重连策略

### 交付物
- 多浏览器同步可用
- 基础重连机制完成

---

## Day 6：版本控制与增量同步
### 目标
避免旧数据覆盖新数据，控制同步冲突。

### 任务
1. conversations/tasks/projects 增加 updated_at 校验策略
2. 评估并补充 version 字段
3. 增量拉取接口设计/补充
4. 前端只刷新变更资源
5. 同步状态字段展示

### 交付物
- 增量同步机制可用
- 冲突覆盖风险降低

---

## Day 7：联调、回归、文档
### 目标
进入稳定版可交付状态。

### 任务
1. 场景 1/2/3 全量回归测试
2. 多账号/多 tab/多浏览器组合测试
3. 平台助手、删除会话、刷新恢复专项测试
4. 关键 store 和同步链路补注释
5. 编写实施说明与运维说明

### 交付物
- 稳定版验收包
- 回归测试记录
- 技术文档

---

## 三、技术决策

### 1. 登录态
- sessionStorage
- 按 tab 隔离

### 2. UI 状态缓存
- sessionStorage
- key 规则：`store:{userId}:{tabId}`

### 3. 业务数据
- 后端数据库为唯一真相
- 前端只做内存镜像
- 页面启动后强制拉取后端数据校正

### 4. 同步机制
- 同浏览器：BroadcastChannel
- 多浏览器：WebSocket user channel

### 5. 同步策略
- 优先增量更新
- 后端事件驱动
- 必要时局部重拉

---

## 四、风险点

1. conversationStore 当前逻辑耦合高，刷新恢复容易回归
2. workspace / kanban / session list 三套状态需要统一口径
3. WebSocket 现有职责偏聊天流，同步事件要谨慎隔离
4. 历史 localStorage/sessionStorage 脏数据需要迁移或废弃版本处理

---

## 五、验收标准

### 场景 1 验收
- 同浏览器 tab A 登录账号 A，tab B 登录账号 B，互不串号

### 场景 2 验收
- 同账号两个 tab 修改会话/任务后，另一个 tab 10 秒内可见
- 同账号不同浏览器修改数据后，另一端自动刷新到新状态

### 场景 3 验收
- 刷新页面后登录态、tab 状态、会话/任务/项目数据均正确恢复

---

## 六、执行约束

- 不在 RC 平台内部新开 admin 会话 ID
- 所有开发同步在当前主会话推进
- 以文档、代码提交、阶段汇报方式同步进度
